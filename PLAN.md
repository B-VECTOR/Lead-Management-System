# LMS — Gap Analysis & Implementation Plan

> Reconciliation of the current codebase against the two **authoritative** reference documents named in `CLAUDE.md`: `LMS_PRD_updated.md` (PRD) and `LMS_Technical_Requirements_updated.md` (Tech Req, **v12**). A third doc, `specs.md` (**v0.6**), is **deprecated** — the frontend was built against it, and it diverges materially from the two updated docs. Per the decisions below, the two updated docs govern; `specs.md` is to be archived.

## 0. Decisions (resolved 2026-07-10)

All previously-open questions are now settled. Recorded here so future sessions don't re-litigate them.

1. **The two updated docs govern.** `specs.md` (v0.6) is old and superseded. The frontend's flattened model (Lead = unit of work, no Project entity, 9-step flow, 3 lead types) is rebuilt to the v12 model: `Company → Lead → Project → Task`, **17-task** BD workflow, `project_details` history, `resource_allocation`, Project-ID generation.
2. **Multi-role via Django Groups — keep.** Users can hold multiple roles (Groups M2M); no single-`role` field.
3. **`username` is restored per the docs.** PRD §5.1 / Tech Req §4.1 list a Username field, so the User model gains a `username` field used as the **login identifier** (SimpleJWT default); `email` remains a required field. *(This changes the currently email-based auth — see Phase 2 note.)*
4. **Keep Attachments, Activity log, and Notifications** in this build (they exist in the frontend today). These are wired to the backend in Phase 8 rather than dropped/deferred. (Finance role remains future scope per PRD §5.10.)
5. **Archive `specs.md`** and fix stale doc references (CLAUDE.md + PRD point at `LMS_Technical_Requirements.md`; the real file is `..._updated.md`). Handled in Phase 0.
6. **Two lead types only: BD and Mining.** Extension is a **workflow cycle inside BD** (Tasks 13–16), not a lead type. Drop the frontend's "Extension" lead type.
7. **`employee_id` and `mobile_no` become numeric (≥0)** per the docs (Phase 2). *(Caveat: numeric `mobile_no` cannot store a leading `+` or formatting — following the docs as instructed.)*
8. **New app for the reference tables** (`countries`, `industries`, `areas`) — not folded into `authentication`.

## 1. Current State Summary

**Working today (verified in code):**
- **Backend auth** — DRF SimpleJWT login/refresh/logout (with rotation + blacklist), self-service password change. Custom email-login `User` model with soft-delete. ([backend/authentication/](backend/authentication/))
- **Backend user management** — full CRUD API (`/api/users/`) gated by a real server-side permission class ([permissions.py](backend/authentication/permissions.py)), plus Django-admin-style template management pages, and a thorough test suite ([tests.py](backend/authentication/tests.py)).
- **RBAC scaffolding** — the 7 roles exist as Django **Groups**, seeded idempotently by `seed_lookups`; `belts` reference table seeded (9 belts). Only the `user_management` group has functional permissions; the other 6 groups are empty placeholders.
- **Frontend shell & auth** — React 19 + Vite + React Query + Tailwind v4 + shadcn/ui. Login, token storage/refresh, route protection, user management screens, and the roles/belts lookups are **wired to the live backend** and functional.
- **Frontend UI (mock-backed)** — a large, polished UI exists for Dashboard, Leads (list/form/detail with task stepper, checklist, files, activity, follow-ups, resources tabs), Follow-ups, Notifications — **but all of it runs on a localStorage mock DB**, not the backend.

**Not working / not built:**
- **The entire lead domain backend** — no `leads`, `tasks`, `checklists`, `workflows`, `resource_allocation`, `project_details`, `followups`, reference-table (country/industry/area) models exist. Only `authentication` app is present ([INSTALLED_APPS](backend/config/settings.py)).
- **The workflow engine** — no task-sequencing logic, no signals, no scheduler/Celery.
- **Everything the frontend shows beyond auth/users is illusory** against the real backend — it reads seeded mock records ([frontend/src/mocks/db.js](frontend/src/mocks/db.js), `api/client.js:5-8`).
- **Server-side RBAC beyond user management** — the 6 non-user-mgmt groups carry no permissions; nothing enforces the PRD §6 matrix server-side.
- **Forgot-password** — mocked in the frontend, no backend endpoint ([auth.js:194-221](frontend/src/api/auth.js#L194-L221)).

**Bottom line:** this is a *frontend-first prototype mid-migration*. Auth + users are cut over to Django; the whole lead-management domain still needs to be built backend-first and then re-wired — and the frontend domain model itself was built to a superseded spec (now being rebuilt to v12).
